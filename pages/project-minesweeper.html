BSc Thesis · 2024 VU Amsterdam

Minesweeper Solver

A browser-based Minesweeper solver that uses the Z3 SMT solver to reason about mine placement. Every deduction is made through formal constraint solving — no guessing, just logic.

GitHub repo Read thesis

What it is

Minesweeper Solver is a browser-based tool that plays Minesweeper using formal constraint reasoning. Rather than applying heuristics or guessing, it encodes the game board as a system of logical constraints and hands them off to the Z3 SMT solver, a theorem prover developed at Microsoft Research. Z3 then determines, with mathematical certainty, which cells are safe to reveal and which must contain mines.

The solver is designed to make its reasoning visible: each step shows which constraints are active, what Z3 concluded, and why. The goal was to build a bridge between an academic tool (Z3) and an interactive, intuitive interface — so that the formal logic happening under the hood becomes legible to anyone watching.

The challenge

The core difficulty was encoding Minesweeper's board state faithfully as SMT constraints. Each revealed cell contributes a numerical clue: the exact number of mines among its neighbours. Translating that into a satisfiable formula — especially as the board evolves move by move — requires careful handling of variable scoping, incremental solving, and conflict resolution when the solver encounters an over-constrained region.

A secondary challenge was connecting the Python-based Z3 backend to a live browser interface. The solver runs step-by-step, pausing to visualise each deduction before proceeding to the next move. Getting that synchronisation right — between the solving loop and the DOM updates — required careful coordination of async state on the JavaScript side.

What I learned

This project taught me what it really means to apply formal methods outside of a textbook. Z3 is a production-grade tool, and using it properly requires understanding how SMT solving works internally — not just its API. I came away with a much stronger intuition for where formal reasoning is genuinely useful versus where it's overkill.

I also learned how much the presentation of a technical system matters. A solver that just outputs answers is a black box. Making the reasoning visible — step by step, in a medium anyone can open in a browser — was the harder design problem, and ultimately the more interesting one.
