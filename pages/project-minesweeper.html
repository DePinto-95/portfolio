BSc Thesis · 2024 VU Amsterdam

An Interactive Minesweeper Solver Using an SMT Solver

A terminal-based Minesweeper solver written in Scala that uses the Z3 SMT solver to reason about mine placement with mathematical certainty. Every deduction is derived from formal constraint solving — no guessing, no heuristics. The solver is ported to JavaScript and playable in the browser below.

Play with solver GitHub repo Read thesis

What it is

This BSc thesis implements an interactive, terminal-based Minesweeper solver in Scala that uses the Z3 SMT solver (Microsoft Research) to reason about mine placement with mathematical certainty. Rather than guessing or applying heuristics, it encodes the board state as a system of integer constraints and runs Z3 via Linux system calls to determine which cells are definitively safe and which are definitively mines.

The program maintains three maps in parallel: the shown map (what the player sees), the full map (the ground-truth board), and the bomb map (Z3 variables: 0 = safe, 1 = mine, −1 = unknown). After each move the solver runs, outputs confirmed safe cells and mine locations, and the player chooses where to click next.

How the solver works

For each opened cell with number n, a constraint is generated: the sum of its unrevealed, unflagged neighbour variables equals n. Z3 finds one satisfying assignment, then — for each frontier cell with value v in that assignment — re-runs with an extra constraint forcing the cell to 1−v. If Z3 returns unsat, the original value is certain. If it finds another solution, the cell remains ambiguous (−1 in the bomb map).

This verify-by-negation loop is the core contribution: it gives provably correct deductions at every step, not just a single satisfying assignment. The solver also handles the unsatisfiable case — when no certain move exists — and tells the player they must guess.

What I learned

Building this forced a deep understanding of how SMT solvers work internally, not just how to call their API. Encoding Minesweeper's rules as integer constraints required careful reasoning about variable scoping, constraint completeness, and incremental solving as the board evolved move by move.

The project also showed me the limits of formal reasoning in games with imperfect information: even a perfect SMT-based solver eventually hits configurations where no certain move exists, and the player must accept probabilistic risk. That boundary between logic and probability turned out to be one of the most interesting parts of the thesis.
